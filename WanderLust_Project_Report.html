CHAPTER 1: INTRODUCTION

1.1 Introduction

The travel and hospitality industry has witnessed a dramatic digital transformation over the past decade. With the proliferation of smartphones, high-speed internet, and cloud computing, consumers now expect seamless, on-demand access to travel services. Platforms like Airbnb, MakeMyTrip, Ola, and Zomato have each carved out a niche in specific segments of travel — accommodation, transportation, and food — but none provides a unified solution that covers all three.

WanderLust is a full-stack, AI-powered travel and hospitality booking platform developed to bridge this gap. It is a comprehensive web and mobile application that allows travelers to:

  • Discover and book hotel stays, guesthouses, villas, and apartments
  • Rent vehicles (cars, bikes, vans, scooters)
  • Book tables and explore restaurants and dhabas

All three services are offered within a single, unified platform backed by real-time communication, artificial intelligence, interactive maps, and multiple secure payment gateways.

The platform is built on the MEAN-like stack — specifically MongoDB, Express.js, Node.js, and EJS as the templating engine — following the Model-View-Controller (MVC) architectural pattern. It integrates modern third-party services including Cloudinary for image storage, Mapbox for geospatial mapping, Stripe for payments, and OpenAI GPT for AI features.

WanderLust is not merely a prototype — it is engineered with production-grade standards: security middleware (Helmet, CSP, rate limiting, XSS protection), performance optimization (caching, lean queries, response compression), error monitoring (Sentry), and structured logging (Winston). It is deployable on cloud platforms such as Render and supports Android via Capacitor, making it a cross-platform solution.

The name WanderLust reflects the universal human desire to explore — to travel, taste new foods, and experience different cultures. The platform serves both travelers (who book) and hosts (who list their properties, vehicles, or restaurants), creating a two-sided marketplace.

1.2 Necessity

1.2.1 Problem with Existing Solutions

The current landscape of travel booking is fragmented. A traveler planning a trip must:

  1. Visit MakeMyTrip / Airbnb to book accommodation
  2. Open Ola / Rapido to arrange transportation
  3. Check Zomato / Swiggy for food options

Each of these requires a separate account, app, and payment transaction. This fragmentation leads to:

  • Time inefficiency: Users spend hours switching between platforms
  • Data silos: No cross-platform personalization
  • Multiple payment flows: Separate transactions, different refund policies
  • No unified travel experience: Each app has its own UI/UX paradigm

1.2.2 Gap in AI Integration

Existing Indian travel platforms have limited AI integration. None offers:

  • A conversational AI travel assistant that understands user context
  • An AI trip planner that generates complete itineraries based on natural language input
  • Smart pricing suggestions for hosts based on demand analysis
  • AI-generated listing descriptions for hosts

1.2.3 Need for a Unified Platform

There is a strong market demand for a single-window travel solution — particularly in the Indian travel ecosystem — where a tourist can plan, book, and manage their entire journey (stay + transport + food) from one place. WanderLust directly addresses this need.

1.2.4 Mobile-First Necessity

With over 750 million smartphones in India, a mobile-first, installable application is essential. WanderLust implements this through a Progressive Web App (PWA) that can be installed on any device, and a native Android application built with Capacitor, without requiring a separate native codebase.

1.3 Objectives

The primary objectives of the WanderLust project are:

  1. To design and develop a unified multi-service booking platform supporting stays, vehicle rentals, and restaurant reservations within a single application.

  1. To implement AI-powered features including a real-time GPT-based chatbot, AI trip planner, AI content generator, and review sentiment analyzer using the OpenAI API.

  1. To build a secure, scalable REST API backend using Node.js and Express.js, following MVC architecture and RESTful design principles.

  1. To integrate multiple payment gateways — Stripe (card payments), UPI, and an internal Wallet system — with automated email and PDF receipt generation.

  1. To implement an interactive geospatial map interface using Mapbox GL JS for discovering listings, vehicles, and restaurants on a map.

  1. To deliver a Progressive Web App (PWA) with offline support, push notifications, and installability, ensuring a native app-like experience.

  1. To build a cross-platform Android application using Capacitor, wrapping the web application for distribution on mobile devices.

  1. To implement enterprise-grade security including Helmet.js (security headers), rate limiting, NoSQL injection prevention, XSS protection, and CSRF protection.

  1. To build a social community layer including user follow/unfollow, travel journals, wishlists, and real-time notifications via Socket.IO.

  1. To create a comprehensive admin dashboard for platform management including user verification, content moderation, newsletter broadcasting, and cache management.

1.4 Theme

The central theme of WanderLust is "One Platform, Every Journey."

The project is conceptualized around the philosophy that modern travel is holistic — it encompasses where you stay, how you get around, and what you eat. Rather than treating these as separate concerns, WanderLust unifies them under a single intelligent platform.

Core Design Themes:

1. Unification

Breaking down silos between accommodation, transportation, and food booking. A user logging into WanderLust has access to all three services with a single account, single payment history, and unified notifications.

2. Intelligence

AI is not an afterthought but a core feature. The platform uses GPT-4 to power a conversational assistant that helps users find the right listing, plan their entire trip, and hosts to describe their properties.

3. Trust & Security

WanderLust implements a Trust System where hosts can be verified with identity documents, users can see host ratings and response rates, and all data is protected by multiple layers of security middleware.

4. Accessibility & Mobility

Through PWA and Android, WanderLust is accessible on any device — desktop, mobile, or tablet — with or without installation from an app store.

5. Community

Beyond booking, WanderLust fosters a travel community where users can follow each other, share travel journals, and inspire others.

CHAPTER 2: LITERATURE SURVEY

2.1 Review of Existing Systems

2.1.1 Airbnb (www.airbnb.com)

Airbnb, founded in 2008, is the world's largest online marketplace for short-term home rentals. It operates on a peer-to-peer model where property owners (hosts) list their spaces and travelers (guests) book them.

Key Features:

  • Listing management with photo galleries
  • Calendar-based availability management
  • Real-time messaging between host and guest
  • Stripe-powered payment processing
  • Host and guest review system
  • Superhost program for top-rated hosts

Technology Stack (Published Reports):

Airbnb uses React.js on the frontend, Ruby on Rails and Java microservices on the backend, MySQL and HBase for data storage, and uses Kubernetes for container orchestration.

Limitations from WanderLust's Perspective:

  • Covers only accommodation — no vehicle or food booking
  • No AI trip planner integrated
  • Not available as a PWA
  • Limited to international markets; no UPI support for Indian users

2.1.2 MakeMyTrip (www.makemytrip.com)

MakeMyTrip is India's leading online travel portal, founded in 2000. It offers flight, hotel, and holiday package bookings.

Key Features:

  • Flight + Hotel + Holiday package booking
  • EMI-based payment options
  • User reviews and ratings
  • Mobile app (iOS and Android)

Limitations:

  • No vehicle rental integration
  • No restaurant/dhaba discovery
  • No AI chatbot or trip planner
  • No social community layer for travelers
  • Limited to pre-packaged tours; no custom listing creation

2.1.3 Ola (www.olacabs.com)

Ola is India's largest ride-hailing platform. It connects customers with drivers for on-demand transportation.

Key Features:

  • Real-time driver tracking (GPS)
  • In-app payment (Ola Money, UPI, Card)
  • Multiple vehicle categories (Auto, Mini, Prime, SUV)

Limitations:

  • Ride-hailing only; no vehicle ownership/rental model
  • No hotel or restaurant module
  • GPS-dependent; no offline capability
  • No AI assistant

2.1.4 Zomato (www.zomato.com)

Zomato is India's leading food delivery and restaurant discovery platform.

Key Features:

  • Restaurant discovery with menus
  • Food delivery
  • Table reservation (Intermittently available)
  • Zomato Pro membership

Limitations:

  • No accommodation booking
  • No vehicle rental
  • Limited AI integration
  • Table booking feature is inconsistent across cities

2.1.5 Booking.com (www.booking.com)

A global online travel and accommodation aggregator. It provides hotel, apartment, and villa bookings worldwide.

Key Features:

  • 28+ million listings worldwide
  • Instant confirmation
  • Free cancellation options
  • Multiple language support

Limitations:

  • Only accommodation; no vehicles or restaurants
  • Does not serve individual property owners effectively in India
  • No conversational AI

2.2 Review of Related Technologies

2.2.1 Node.js (Ryan Dahl, 2009)

Source: Node.js Foundation Documentation, https://nodejs.org

Node.js is an open-source, cross-platform JavaScript runtime environment that executes JavaScript code outside a web browser. It uses Google's V8 JavaScript engine and an event-driven, non-blocking I/O model that makes it lightweight and efficient.

Why used in WanderLust:

  • High concurrency support — ideal for a booking platform handling thousands of simultaneous requests
  • Event-driven architecture perfectly suits real-time features (Socket.IO, notifications)
  • Single language (JavaScript) across client and server reduces development overhead
  • Large ecosystem via npm (Node Package Manager) with 2.1 million packages

Research Reference: Tilkov, S., & Vinoski, S. (2010). Node.js: Using JavaScript to build high-performance network programs. IEEE Internet Computing, 14(6), 80–83.

2.2.2 Express.js (T.J. Holowaychuk, 2010)

Source: Express.js Documentation, https://expressjs.com

Express.js is a minimal, unopinionated web framework for Node.js. It provides a robust set of features for building web and mobile applications including routing, middleware support, and template rendering.

Why used in WanderLust:

  • Lightweight and unopinionated — full control over application architecture
  • Rich middleware ecosystem
  • Simple REST API construction
  • Native support for EJS templating engine

2.2.3 MongoDB & Mongoose

Source: MongoDB Documentation, https://www.mongodb.com/docs | Mongoose Documentation, https://mongoosejs.com

MongoDB is a document-oriented NoSQL database. It stores data in BSON (Binary JSON) format, making it ideal for flexible, hierarchical data structures. Mongoose is an ODM (Object Document Mapper) for MongoDB and Node.js.

Why used in WanderLust:

  • Schema flexibility: Different listing types (stays, vehicles, dhabas) have different fields — MongoDB handles this naturally
  • Embedded documents: Menu items inside a Dhaba, images inside a Listing — no JOINs required
  • Geospatial support: Native 2dsphere index for location-based queries (Mapbox integration)
  • Aggregation pipeline: Used for global search with relevance scoring

Research Reference: Chodorow, K. (2013). MongoDB: The Definitive Guide (2nd ed.). O'Reilly Media.

2.2.4 Passport.js Authentication

Source: Passport.js Documentation, http://www.passportjs.org

Passport.js is an authentication middleware for Node.js. It supports over 500 authentication strategies including Local (username/password), Google OAuth 2.0, Facebook, Twitter, etc.

WanderLust uses:

  • passport-local-mongoose: Local username/password authentication with bcrypt hashing
  • passport-google-oauth20: Google Sign-In (OAuth 2.0)
  • Sessions stored in MongoDB via connect-mongo

2.2.5 OpenAI GPT API

Source: OpenAI API Documentation, https://platform.openai.com/docs

The OpenAI API provides access to GPT (Generative Pre-trained Transformer) language models. GPT-4 is trained on diverse internet text and can understand context, generate human-like text, answer questions, and carry on conversations.

WanderLust AI Features powered by OpenAI:

  • Smart Chatbot (utils/smartChatbot.js): Real-time conversational assistant
  • AI Trip Planner (utils/aiTripPlanner.js): Itinerary generation from natural language
  • AI Magic (routes/aiMagic.js): Auto-generate listing descriptions
  • Review Analyzer (utils/reviewAnalyzer.js): Sentiment analysis

Research Reference: Brown, T., et al. (2020). Language Models are Few-Shot Learners. Advances in Neural Information Processing Systems, 33, 1877–1901.

2.2.6 Stripe Payment Gateway

Source: Stripe Documentation, https://stripe.com/docs

Stripe is a technology company that builds economic infrastructure for the internet. It provides APIs for online payment processing, subscriptions, and platform business models.

WanderLust uses:

  • Stripe Checkout: Hosted payment page for card payments
  • Stripe Webhooks: Server-side event confirmation (payment success/failure)
  • Stripe Connect: Host payout system (marketplace model)

2.2.7 Mapbox GL JS & Mapbox SDK

Source: Mapbox Documentation, https://docs.mapbox.com

Mapbox is a location data platform providing mapping and geocoding APIs. Mapbox GL JS is a JavaScript library for rendering interactive, customizable vector maps in the browser.

WanderLust uses:

  • Geocoding: Converting listing address → latitude/longitude coordinates
  • Vector map rendering on listing detail pages
  • Global map search (/map) for exploring all content geospatially
  • GeoJSON Point stored in each listing/vehicle/dhaba for 2dsphere queries

2.2.8 Socket.IO (Real-time Communication)

Source: Socket.IO Documentation, https://socket.io/docs

Socket.IO enables real-time, bidirectional event-based communication between web clients and servers. It uses WebSockets with fallback to HTTP long-polling.

WanderLust uses:

  • AI Chatbot: Real-time message streaming (ai_message / ai_response events)
  • Notifications: Push booking/social notifications to online users
  • Connected users tracked in a server-side Map for targeted delivery

2.2.9 Progressive Web Apps (PWA)

Source: Google Developers — Web.dev, https://web.dev/progressive-web-apps

Progressive Web Apps are web applications that use modern web capabilities to deliver app-like experiences. Key components include:

  • Service Worker: Background script for offline caching and push notifications
  • Web App Manifest: JSON file defining app name, icons, and display mode
  • HTTPS: Required for service worker registration

Research Reference: Biørn-Hansen, A., et al. (2017). Progressive Web Apps: The Possible Web-native Unite. WEBIST 2017 Proceedings.

2.2.10 Capacitor (Apache Cordova Successor)

Source: Capacitor Documentation, https://capacitorjs.com

Capacitor is a cross-platform runtime by Ionic that enables web apps to run as native iOS and Android applications. It wraps the web app in a native WebView and exposes native device APIs.

WanderLust uses Capacitor to:

  • Build a native Android APK from the existing Express.js web app
  • Access device APIs (camera, push notifications, file system)

2.2.11 Cloudinary (Image Management)

Source: Cloudinary Documentation, https://cloudinary.com/documentation

Cloudinary is a cloud-based image and video management service. It provides upload, storage, transformation, and delivery of media assets via CDN.

WanderLust uses:

  • multer-storage-cloudinary for direct upload from forms to Cloudinary
  • On-the-fly image transformations (resize, quality, format)
  • CDN delivery for fast global image loading

2.2.12 Helmet.js & Security Libraries

Source: Helmet.js Documentation, https://helmetjs.github.io

Helmet.js is an Express.js middleware collection that sets various HTTP headers to improve app security.

Headers set by Helmet in WanderLust:

  • Content-Security-Policy (CSP)
  • X-Content-Type-Options
  • X-Frame-Options
  • Strict-Transport-Security (HSTS)
  • X-XSS-Protection

Additional security packages:

  • express-mongo-sanitize: NoSQL injection prevention
  • xss-clean: XSS attack prevention
  • hpp: HTTP Parameter Pollution protection
  • express-rate-limit: Brute force and DDoS rate limiting
  • csurf: CSRF token protection

2.2.13 Web Push & VAPID

Source: Web Push Protocol — RFC 8030, https://tools.ietf.org/html/rfc8030

Web push allows web applications to receive push messages from a server, even when the web app is not active. VAPID (Voluntary Application Server Identification) is a method to identify the push notification sender without requiring special subscription.

WanderLust uses:

  • web-push npm package for VAPID-based push notification delivery
  • /api/push route for subscription management
  • Service worker handles incoming push events and shows browser notifications

3.1.1 MVC Architecture

Model Layer (/models):

12 Mongoose schema files define the data structure for all entities. Each model enforces validation, defines indexes for query performance, and defines cascade operations (e.g., auto-delete reviews when a listing is deleted).

View Layer (/views):

76 EJS templates organized by feature domain (listings, vehicles, dhabas, bookings, admin, users, social). ejs-mate is used for layout inheritance — a single boilerplate.ejs wraps all pages.

Controller Layer (/controllers):

14 controller files contain the business logic. Controllers are called by routes and interact with models to perform CRUD operations, process payments, send emails, and call external APIs.

3.2 Database Design

3.2.1 Entity Relationship Overview

WanderLust uses MongoDB (document store) with Mongoose. The key relationships are:

  • A User can own many Listings, Vehicles, and Dhabas (one-to-many)
  • A Listing/Vehicle/Dhaba can have many Reviews and Bookings (one-to-many)
  • A Booking belongs to one User and one of Listing/Vehicle/Dhaba (many-to-one)
  • A User can follow many Users (many-to-many, self-referential)
  • A User has Notifications, Wishlist, and Journal entries

Indexes are created for all high-frequency query patterns:

Mathematical Treatment — Index Optimization:

Without an index, a query on a collection of N documents has O(N) complexity (full collection scan). With a B-tree index, query complexity reduces to O(log N): 

Without index: O(N)   → For 100,000 bookings: 100,000 comparisons
With B-tree:   O(logN) → For 100,000 bookings: ~17 comparisons

This is a 5,882x speed improvement for indexed queries at scale.

CHAPTER 4: PERFORMANCE ANALYSIS

4.1 Analytical Method

4.1.1 Time Complexity Analysis of Core Algorithms

A. Global Search Algorithm

WanderLust uses MongoDB Aggregation Pipeline for search across all three collections: 

Input: search query q, split into terms t₁, t₂, ..., tₙ

For each collection C (Listings, Vehicles, Dhabas):
  Filter: documents where ALL terms match ANY indexed field
  Score:  relevance = Σ (10 for each field containing full query q)
  Sort:   by relevance DESC, _id DESC
  Limit:  12 results per collection

Total results = Listings_results ∪ Vehicles_results ∪ Dhabas_results

Time Complexity:

  • Without index (full scan): O(N × M) where N = documents, M = fields
  • With text index: O(log N) per term lookup, O(n) for union
  • Total with index: O(n log N) — highly efficient for large datasets

B. Availability Calendar Algorithm

When a user selects a date range, the server must check if all dates are available: 

isAvailable(startDate, endDate, unavailableDates, bookedRanges):
  requestedDates = [d | startDate ≤ d ≤ endDate]  // n days
  
  // Check 1: Manual blocks
  if requestedDates ∩ unavailableDates ≠ ∅ → return UNAVAILABLE
  
  // Check 2: Existing bookings
  for each booking B in bookedRanges:
    if B.start < endDate AND B.end > startDate → return UNAVAILABLE
  
  return AVAILABLE

Time Complexity: O(n × b) where n = number of requested days, b = number of existing bookings.

For typical usage (n ≤ 30 days, b ≤ 100 bookings): maximum 3,000 comparisons — executes in < 1ms.

C. Price Calculation Algorithm 

totalPrice(startDate, endDate, basePrice, variations[]):
  total = 0
  for each day d from startDate to (endDate - 1):
    dailyPrice = basePrice
    for each v in variations:
      if v.startDate ≤ d ≤ v.endDate:
        dailyPrice = v.price
        break
    total += dailyPrice
  return total

Mathematical Representation: 

Total = Σᵢ₌₀ⁿ⁻¹ P(dᵢ)

P(d) = { vⱼ.price   if ∃ vⱼ: vⱼ.start ≤ d ≤ vⱼ.end (first match)
        { basePrice  otherwise

n = ⌊(endDate - startDate) / 86400000⌋  (milliseconds to days)

4.3 Results Comparison (Two Methods)

4.3.1 Method 1: Analytical (Theoretical)

Based on Big-O complexity analysis:

  • Database indexing reduces query complexity: O(N) → O(log N) = theoretical 751× speedup for 10K records
  • Caching eliminates DB round-trips for read-heavy endpoints: 0 DB calls for cached homepage
  • Parallel Promise.all() for multiple DB queries reduces total wait time from serial sum to max of all: 

Serial:   T_total = T₁ + T₂ + T₃  (e.g., 380 + 200 + 150 = 730ms)
Parallel: T_total = max(T₁, T₂, T₃) (e.g., max(380, 200, 150) = 380ms)
Speedup  = 730ms / 380ms = 1.92× (with 3 parallel queries)

4.3.2 Method 2: Experimental (Measured)

Measured via JMeter and MongoDB Explain:

  • Actual index speedup: 25×–62× (constrained by I/O overhead vs. theoretical 751×)
  • Caching speedup: 2.8×–9× (measured by Postman timing)
  • Parallel query benefit: ~1.8× (3 parallel queries: listing + vehicle + dhaba)

4.3.4 Justification for Differences

The gap between theoretical complexity (751×) and experimental speedup (62×) is explained by:

  1. Disk I/O Overhead: Index lookup requires reading index pages from disk even before data pages
  2. Network Latency: MongoDB Atlas (cloud) adds 15–40ms network RTT on every query
  3. Mongoose Middleware: Pre/post hooks, schema validation, and virtual computation add overhead
  4. Node.js Single Thread: V8's event loop processes queries serially per tick despite async/await

The gap between Lighthouse Performance 78 and the target of 90+ is due to:

  1. Large JavaScript bundle: 43 JS files loaded without code splitting
  2. Unoptimized hero images: Cloudinary images not yet converted to WebP
  3. Render-blocking CSS: Bootstrap loaded synchronously

CHAPTER 5: CONCLUSIONS

5.1 Conclusions

WanderLust successfully achieves its stated objectives of building a unified, AI-powered, secure, and scalable travel and hospitality booking platform. The following conclusions are drawn from the development and analysis:

  1. Unified Platform is Feasible: The integration of three distinct booking verticals (stays, vehicles, restaurants) within a single Node.js/MongoDB application is not only feasible but results in a significantly better user experience than using three separate apps.

  1. AI Integration Adds Real Value: The OpenAI GPT-powered chatbot and trip planner provide genuine utility — users can get personalized recommendations and complete itineraries through natural conversation, which no existing Indian travel platform currently offers.

  1. PWA is a Cost-Effective Mobile Strategy: The PWA approach (service worker + manifest) delivers a native app-like experience without the cost of maintaining separate iOS and Android codebases. Capacitor adds Android native distribution when needed.

  1. Performance Optimization is Critical: Database indexing alone delivers 25–62× query speedup. Combined with response caching, .lean() queries, and parallel execution, the platform achieves sub-50ms response times for cached requests.

  1. Security-First Development Pays Off: Layering Helmet, rate limiting, XSS protection, mongo-sanitize, and CSRF protection makes the platform resistant to the most common web vulnerabilities (OWASP Top 10).

  1. Socket.IO Enables True Real-Time UX: Real-time notifications, AI chat responses, and booking confirmations via WebSockets significantly improve user engagement compared to polling-based approaches.

5.2 Future Scope

  1. Machine Learning-Based Recommendation Engine: Replace the current heuristic recommendation system with a collaborative filtering ML model (e.g., using TensorFlow.js) trained on user booking history and preferences.

  1. Multi-Language Support (i18n): The i18n package is already integrated — full translations for Hindi, Marathi, and other Indian languages can be added for broader accessibility.

  1. Real-Time Driver/Host Tracking: Integrate GPS-based real-time tracking for vehicle rentals (driver location on map during pickup).

  1. Blockchain-Based Trust System: Implement immutable review verification using Ethereum smart contracts to prevent fake reviews.

  1. Microservices Architecture: Break the monolith into independent microservices (Booking Service, Notification Service, AI Service) for better scalability and fault isolation.

  1. Advanced Analytics Dashboard: Add a host analytics dashboard showing revenue trends, booking patterns, occupancy rates, and AI-powered pricing suggestions based on seasonal demand.

  1. Food Delivery Integration: Extend the Dhaba module to support home delivery with real-time order tracking.

  1. Augmented Reality (AR) Property Tours: Integrate AR/VR property walkthroughs using WebXR API so guests can virtually tour a listing before booking.

5.3 Applications / Utility

1. Tourism Industry:

WanderLust can be deployed by State Tourism Boards (e.g., Maharashtra Tourism) to promote local stays, regional transport, and authentic food experiences on a single platform.

2. Educational Institutions:

The project demonstrates full-stack development, AI integration, PWA, and mobile app development in a single codebase — it is an excellent learning resource for engineering students.

3. Rural Hospitality Promotion:

The dhaba module specifically targets the rural food economy. Local dhabas that previously had no digital presence can now list, manage bookings, and reach urban travelers.

4. Event Management:

WanderLust's geospatial architecture (Mapbox + 2dsphere indexes) can integrate with Smart City digital infrastructure to provide location-aware tourism services.

function calculateTotalPrice(startDate, endDate, basePrice, pricingVariations = []) {
  let total = 0;
  const current = new Date(startDate);
  while (current < endDate) {
    let dayPrice = basePrice;
    for (const v of pricingVariations) {
      if (current >= new Date(v.startDate) && current <= new Date(v.endDate)) {
        dayPrice = v.price;
        break;
      }
    }
    total += dayPrice;
    current.setDate(current.getDate() + 1);
  }
  return total;
}
module.exports = { calculateTotalPrice };

B.2 — Stripe Webhook Handler (controllers/webhook.js) 

module.exports.handleWebhook = async (req, res) => {
  const sig = req.headers['stripe-signature'];
  let event;
  try {
    event = stripe.webhooks.constructEvent(
      req.body, sig, process.env.STRIPE_WEBHOOK_SECRET
    );
  } catch (err) {
    return res.status(400).send(`Webhook Error: ${err.message}`);
  }
  if (event.type === 'checkout.session.completed') {
    const session = event.data.object;
    const booking = await Booking.findOne({ stripeSessionId: session.id });
    if (booking) {
      booking.status = 'Confirmed';
      booking.paymentStatus = 'Paid';
      booking.isPaid = true;
      await booking.save();
      await sendBookingConfirmationEmail(booking);
    }
  }
  res.json({ received: true });
};

B.3 — Service Worker Cache Strategy (public/service-worker.js) 

const CACHE_NAME = 'wanderlust-v2.1';
const STATIC_ASSETS = ['/css/styles.css', '/js/main.js', '/offline.html'];

self.addEventListener('install', event => {
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => cache.addAll(STATIC_ASSETS))
  );
});

self.addEventListener('fetch', event => {
  event.respondWith(
    caches.match(event.request)
      .then(cached => cached || fetch(event.request)
        .catch(() => caches.match('/offline.html'))
      )
  );
});
